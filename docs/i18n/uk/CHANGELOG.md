# Changelog

Усі помітні зміни в BgRemover задокументовано в цьому
файлі. Формат орієнтується на
[Keep a Changelog](https://keepachangelog.com/de/1.1.0/); проєкт
дотримується [Semantic Versioning](https://semver.org/lang/de/).

## [Unreleased]

### Змінено

- **Нотатки релізу GitHub тепер беруться з CHANGELOG.** Робочий процес
  публікації (`release-linux.yml`) виводить тіло релізу для тега `vX.Y.Z` із
  розділу `## [X.Y.Z]` файлу `CHANGELOG.md` і передає його через `--notes-file`
  до `gh release` — зокрема й при повторному використанні наявного релізу
  (`gh release edit`), а не лише під час першого створення. Жорстко зашитий
  текст «Automated build…» зникає; якщо відповідного розділу немає, завдання
  публікації явно завершується з помилкою (без тихого запасного варіанта)
  (#311).
- **Щотижневий бенчмарк більше не повідомляє про артефакти середовища як про
  регресії.** Кожен результат (`benchmarks/results/`) тепер містить відбиток
  середовища (версія Python/Pillow/NumPy, архітектура, кількість CPU, раннер);
  порівняння пропускає непорівнянні базові лінії (відсутній відбиток, інші версії
  чи параметри бенчмарку) і підтверджує підозрілий результат у тому самому
  запуску через кілька повторень (медіана), перш ніж створити issue
  (#277, #278, #279).

## [2.4.1] – 2026-06-17

### Виправлено

- **Завантажуваний застосунок для macOS (`.dmg`) після запуску безкінечно
  відкривав нові вікна.** У замороженому бандлі AI-інференс запускає дочірній
  процес через multiprocessing «spawn», що перезапускає сам бінарний файл
  застосунку; без `multiprocessing.freeze_support()` у точці входу бандла кожен
  дочірній процес знову запускав GUI → «fork bomb» зі 100+ вікнами, яку зупиняв
  лише перезапуск. Точка входу PyInstaller тепер спершу викликає
  `freeze_support()`, тож дочірній процес інференсу стартує коректно замість
  відкривати GUI.

- **Завантажуваний застосунок для macOS (`.dmg`) не запускався.** Заморожений
  бандл аварійно завершувався вже на `import bgremover` з `PackageNotFoundError`,
  а потім `FileNotFoundError`, бо PyInstaller не включав метадані пакета, а в
  бандлі немає `pyproject.toml` як запасного варіанта — піктограма лише коротко
  блимала, після чого нічого не відбувалося. Spec для PyInstaller тепер вбудовує
  метадані `*.dist-info` (`copy_metadata`), а визначення версії більше не може
  перервати запуск (захисний запасний варіант замість необробленого винятку).

- **Видалення фону через ШІ у `.dmg` не завантажувалося.** Дочірній процес
  інференсу помирав на `import rembg` з `PackageNotFoundError` («No package
  metadata was found for pymatting»): PyInstaller пакує код залежностей rembg,
  але не їхні метадані `*.dist-info`, а `pymatting` під час імпорту читає власну
  версію. Spec тепер вбудовує метадані всього ланцюга залежностей rembg
  (`copy_metadata(…, recursive=True)`).

## [2.4.0] – 2026-06-15

### Додано

- **Застосунок macOS для завантаження (`.dmg`).** Самодостатній пакунок
  `BgRemover.app` (PyInstaller, Apple Silicon/arm64) збирається як `.dmg` і
  додається до релізу GitHub — аналогічно до Linux-AppImage та без локального
  встановлення Python. Пакунок наразі **без підпису**: під час першого запуску
  відкрийте його через правий клік → «Відкрити». Збірка через
  `packaging/mac/build_macos.sh`.
- **Артефакти завантаження тепер містять ШІ-вилучення тла.** Linux-AppImage і
  macOS-`.dmg` містять `rembg`/`onnxruntime`, тож ШІ в один клік працює без
  додаткового встановлення (артефакти відповідно більші).
- **Робочий процес релізу збирає кросплатформно.** `release-linux.yml` тепер,
  окрім Linux-AppImage та `.deb` (x86_64 + aarch64/Raspberry Pi OS), створює ще
  й macOS-arm64-`.dmg` для тега `vX.Y.Z` і публікує всі артефакти разом.
- **Відкриття зображень через асоціації файлів і командний рядок.**
  `bgremover image.png` та `python -m bgremover image.png` відкривають шлях після
  побудови вікна тим самим валідованим асинхронним шляхом завантаження, що й
  діалог файлів, Нещодавні та drag & drop; також обробляються Linux-desktop-запис
  (`%F`) і macOS-`QFileOpenEvent` (Finder «Відкрити за допомогою», подвійний клік).
  Кілька шляхів: перший відкривається, решта ігнорується із зазначенням кількості
  в рядку стану; відсутні, непідтримувані чи нелокальні шляхи відхиляються
  контрольовано замість переривання запуску, а перед заміною відредагованого
  зображення спрацьовує запит про незбережені зміни. Під час виходу із застосунку
  робочі потоки додатково коректно завершуються (#249).
- **Бенчмарк продуктивності конвеєра обробки зображень.**
  `scripts/benchmark.py` вимірює час обробки для кожного вихідного формату
  (PNG/JPEG/WebP/TIFF) реальними шляхами `image_ops`, зберігає датовані
  результати в `benchmarks/results/` і порівнює послідовні запуски; формати з
  регресією понад 10 % позначаються та за потреби повідомляються як issue на
  GitHub (`make bench` / `make bench-compare`). Щотижневий CI-workflow
  (`.github/workflows/benchmark.yml`) виконує запуск і порівняння на сталому
  обладнанні та записує результат назад як наступну базову лінію.
- **Поведінкові тести посилено.** Розширено покриття поведінковими тестами для
  раніше неповних шляхів (#177, #192).
- **Окремі юніт-тести для `app.py` і `main_window.py`.** Покриття `app.py`
  0 % → 100 % і `main_window.py` 68 % → 100 %; загальне покриття зросло до
  94 % (#214).

### Змінено

- **Оновлено залежності.** `idna` піднято до 3.15, а `urllib3` до 2.7.0;
  `LICENSES.md` синхронізовано з новим dependency snapshot.
- **Складальний бекенд закріплено проти CVE ланцюга постачання.** `setuptools`
  піднято до `>=78.1.1` у `pyproject.toml` (`[build-system]`) та
  `requirements/constraints.txt` (CVE-2024-6345 RCE, CVE-2025-47273 path
  traversal), а `wheel` до `==0.46.2` у `constraints.txt` (CVE-2026-24049). Тож
  ізольований збір wheel більше не може підтягнути вразливі складальні
  інструменти (#200, #201).
- **pip піднято до виправленої версії в CI/dev.** CI-воркфлоу, що
  встановлюють через pip (`ci.yml`, `pr-ci.yml`, `ui-nightly.yml`,
  `benchmark.yml`, `license-check.yml`), та веб-хук SessionStart піднімають
  `pip` до `>=26.1.2` перед встановленням, так само як dev-документація зі
  встановлення (`README.md`/`INSTALL_MAC.md`/`INSTALL_LINUX.md`). Закриває батч
  `pip-audit` із CVE path traversal, symlink і захоплення модулів; pip — це сам
  інструмент встановлення, тож його не можна закріпити через
  `constraints.txt` (#202).
- **Діагностика macOS редагує чутливі шляхи.** `diagnose_mac.sh` тепер типово
  замінює `$HOME` на `~`, скорочує решту шляхів `/Users/<name>` і замість
  останніх 40 сирих рядків логу виводить відфільтроване зведення помилок із
  редагованими шляхами — вивід можна безпечно додавати до баг-репортів. Повну
  діагностику (разом із сирим логом) дає новий прапорець
  `--include-raw-logs`; shell-тест (`tests/test_diagnose_mac.py`) гарантує,
  що домашній каталог і шляхи зображень не потрапляють до типового
  виводу (#185).
- **Залежності релізу AppImage закріплено.** Snapshot
  `requirements/constraints.txt` фіксує версії для AppImage-білд-воркфлоу
  (#182, #191).
- **Дозволи license-воркфлоу посилено.** Воркфлоу тепер виконується з
  мінімальними правами (#183, #193).
- **`CanvasHistory._redo_max` видалено.** Атрибут лише для запису ніде не
  читався; обмеження redo забезпечується винятково через `deque(maxlen=…)`
  (#199, #215).
- **`import bgremover` більше не завантажує Qt-стек.** Точка входу пакета
  (`bgremover/__init__.py`) тепер напряму експортує лише легкі метадані
  (`__version__`, `get_version`); усталені GUI-/Qt-ре-експорти (`ImageCanvas`,
  `MainWindow`, workers …) лишаються сумісними, але завантажуються ліниво через
  `__getattr__` PEP 562 під час першого доступу до атрибута. Запити версії та
  метаданих тепер працюють headless без PyQt6; регресійний тест у підпроцесі
  гарантує, що легкий імпорт не тягне ні `bgremover.canvas`/`main_window`, ні
  PyQt6 у `sys.modules` (#232).

### Виправлено

- **Підпроцес rembg посилено (надійність і пам'ять).** Чотири подальші висновки
  з Codex-рев'ю #283 у `bgremover/ai_process.py`: після тимчасової помилки
  `new_session()` сесія rembg відбудовується — рівно один раз — на наступному
  запиті, замість переходу на `remove(..., session=None)` із перезавантаженням
  моделі на кожному виклику (гарантія #229 збережена); неактивний дочірній
  процес одразу звільняє останній вхідний PNG, а не утримує його; вхідний і
  результуючий PNG передаються як сирі байтові кадри (`send_bytes`/`recv_bytes`),
  а не серіалізуються через pickle в каналі, що усуває піки пам'яті та ризик OOM
  на великих зображеннях (до 40 МП); а `request_stop()`, що надходить саме під
  час запуску процесу, переноситься на свіжий процес через пару
  `_proc_lock`/`_stop_pending`. Регресійні тести покривають усі чотири шляхи
  (#285).
- **Піки пам'яті в обмеженому читанні файлу зменшено.** Два подальші висновки з
  Codex-рев'ю #264 у `bgremover/image_loading._read_capped`: замість
  `b"".join(chunks)` (що тримав чанки **і** результат одночасно, ~1 ГіБ біля
  ліміту 512 МіБ) вміст збирається в один заздалегідь розмірений `bytearray` і
  передається напряму — більше немає піку ~2×. Перший read до того ж обмежений
  розміром, відомим через `os.fstat()`, тож малий файл більше не запитує ~8 МіБ
  запасу; невеликий наступний read і надалі виявляє ріст між `fstat()` і
  читанням (TOCTOU) чи ненадійний `st_size` (pipes/sockets). Виявлення
  ліміту/перевищення (`None`) незмінне; регресійні тести покривають обидва шляхи
  (#286).
- **Ліміт розміру вхідного файлу перед читанням.** `open_validated_image`
  тепер перевіряє вхідний файл через `os.fstat()` проти задокументованого
  байтового ліміту (`_MAX_INPUT_FILE_BYTES`, 512 МБ) **перш ніж** його вміст
  повністю читається в пам'ять; додатковий обмежений `read()` захищає від
  незвичних файлових об'єктів і зміни розміру між `fstat()` та `read()`
  (TOCTOU). Повідомлення розрізняє розмір файлу (МБ) і ліміт мегапікселів (MP).
  Синхронний та асинхронний шляхи завантаження використовують ту саму
  перевірку; наявний ліміт мегапікселів і захист TOCTOU збережено (#230).
- **Сесія інференсу rembg повторно використовується.** Warmup тепер створює
  рівно одну rembg/ONNX-сесію через `new_session()` і зберігає її на рівні
  модуля; кожен подальший `AIWorker` передає її в `remove(..., session=...)`
  замість повторної ініціалізації моделі. Створення потокобезпечне завдяки
  double-checked locking і виконується щонайбільше один раз упродовж кількох
  KI-викликів; невдала ініціалізація все одно повідомляє помилку worker і не
  лишає хибно «готового» стану. Оманливий коментар (нібито фіктивний `remove()`
  кешує сесію) виправлено принагідно (#229).
- **`recent_files` стійкий до пошкоджених налаштувань.** `RecentFiles.paths()`
  тепер обробляє будь-який збережений сирий тип захищено: одиночний рядок
  лишається одним записом, списки/кортежі фільтруються поелементно до
  непорожніх рядків, а будь-яке інше значення (напр. ціле число, `None`) дає
  порожній список замість `TypeError`. Новий `sanitize()` одноразово під час
  старту перезаписує справді пошкоджене значення в очищеному вигляді (з
  попередженням у лог); нешкідливий одноелементний рядок QSettings лишається
  недоторканим. Тож вручну змінене або застаріле значення `recent_files` більше
  не перериває побудову меню чи застосунку; новіша (майбутня) схема також
  лишається незайманою, щоб уникнути втрати даних при downgrade (#233, #240).
- **Double-checked lock для лінивого імпорту rembg і TOCTOU-захист у
  `open_validated_image`.** Два потоки могли одночасно ввійти в імпорт (race), а
  файл відкривався двічі (вікно TOCTOU); обидва випадки покрито регресійними
  тестами (#174).
- **Застарілі результати асинхронного завантаження зображень відкидаються.**
  Монотонний лічильник `_load_generation` у `MainWindow` не дає пізньому
  load-колбеку перезаписати новіше зображення (аналогічно AI-stale-перевірці)
  (#190).
- **Типізацію маски виділення canvas виправлено.** Неправильний тип спричиняв
  помилку mypy у повному CI-прогоні (#196, #197).
- **YAML CI-воркфлоу полагоджено.** Незакавичена назва кроку оновлення pip
  ламала парсинг воркфлоу (#213).
- **Активний кроп більше не переживає зміну стану зображення.** Кожна видима
  зміна зображення (поворот, віддзеркалення, результат ШІ, undo/redo,
  відновлення оригіналу, підтвердження кропу) тепер централізовано в
  `_set_image_state` відкидає активний оверлей кропу та розпочате ласо й
  випромінює `cropModeChanged(False)` рівно один раз. Тож застарілий прямокутник
  кропу більше не можна застосувати до нового зображення, і він не створює
  прозорих padding-пікселів (#247).
- **Реліз-воркфлоу публікує лише після зеленого Full-CI-гейту.**
  `release-linux.yml` тепер викликає головну Full-CI-матрицю (`ci.yml`) як
  повторно використовуваний воркфлоу й прив'язує build і publish до неї через
  `needs`; окремий job `verify-tag` падає, якщо тег не відповідає формату
  `vX.Y.Z` або розходиться з `project.version`. AppImage/`.deb` перед
  завантаженням перевіряються за назвою, архітектурою, виконуваністю та
  Debian-метаданими, а помилки `gh release create` більше не ховаються за
  `|| true` (наявний реліз явно повторно використовується). Тож жоден артефакт із
  коміту з червоними тестами чи розбіжною версією більше не потрапляє в
  реліз (#250).
- **Порожнє виділення одразу звільняє pixmap оверлею.** `_refresh_overlay`
  тепер перевіряє порожній стан маски **перед** інкрементним dirty-шляхом. Коли
  гумка стирає останній виділений піксель, `_overlay_pixmap` і
  `QGraphicsPixmapItem` негайно очищаються, замість того щоб тримати прозору
  повнорозмірну QPixmap (близько 160 МіБ за 40 MP) до наступної повної
  перебудови. Часткове стирання й далі оновлює лише змінений прямокутник (#251).
- **Доопрацювання реліз-воркфлоу посилено.** Job publish тепер задає `GH_REPO`,
  щоб `gh release` звертався до правильного репозиторію без checkout; повторно
  використовуваний job test залежить від `verify-tag`, тож недійсний або
  невідповідний версії тег більше не запускає матрицю; а `download-artifact`
  завантажує артефакти через `run-id`/`github-token` (з `actions: read`) з усього
  run, тож «Re-run failed jobs» більше не втрачає артефакти попередньої спроби.
  README/RESOURCES (зокрема переклади) більше не описують видалений тригер
  `release: published` (#257).
- **Ліміт завантаження зображення без преалокації 512 МіБ і локалізований.**
  `open_validated_image` тепер читає вміст файлу блоками по 8 МіБ (замість
  `read(limit + 1)`, який у буферизованому читачі CPython одразу резервує
  ~512 МіБ і може спричинити `MemoryError` для маленького файлу при обмеженій
  памʼяті); зростання між `fstat()` і читанням і далі виявляється через
  `limit + 1`. Повідомлення про розмір іде через ключ перекладу
  `status.file_too_large` (повністю локалізовано de/en замість змішаного
  повідомлення) і округлює фактичне значення вгору, а ліміт — вниз, тож воно
  помітно більше за „ліміт + 1 байт“ (напр. „513 MB“ за максимуму „512 MB“,
  замість обох „512 MB“ із `.0f`) (#258).
- **Міграція схеми QSettings безпечна при downgrade.** Відсутня міграція більше
  не піднімає `schema_version` до поточного значення без перевірки, а вища
  майбутня схема більше не перезаписується під час побудови меню нещодавніх
  файлів — випадковий downgrade тому не втрачає налаштувань (#234, #259).
- **Escape спершу скасовує розпочате ласо; курсор інструмента відновлено після
  обрізки.** Розпочате полігональне ласо тепер скасовується Escape перед
  очищенням виділення (порядок обрізка > ласо > виділення). Коли активна обрізка
  скасовується автоматично, `_finish_mode` відновлює курсор активного
  інструмента замість того, щоб лишати курсор обрізки (#248, #260).
- **Зупинка workers обмежена в часі.** Під час закриття застосунку
  `WorkerController` тепер чекає лише обмежено на `quit()`/`wait()`, перш ніж
  вдатися до `terminate()` з повторним обмеженим `wait()`; worker, що не
  відповідає, більше не блокує закриття безмежно, а шлях помилки логиться.
  Власне ризик `terminate()` для нативної ONNX-роботи згодом усунено,
  перенісши інференс rembg/ONNX в окремий процес, запущений через `spawn`
  (`ai_process`): воркер ШІ лише опитує результат і кооперативно зупиняється,
  скасування та закриття жорстко завершують процес інференсу, а `terminate()`
  більше не є аварійним виходом для роботи ШІ (#270, наслідок #231).
- **Overlay пензля уникає повного сканування маски на кожен рух миші.**
  `canvas_selection` веде лічильник виділення інкрементально й використовує
  bounding box зміни замість сканування всієї маски на кожен рух пензля/гумки;
  `has_selection` тому O(1). Це тримає великі зображення плавними під час
  швидкого малювання (#261).

### Видалено

- **Видалено мертвий код (#244).** Ніде не викликаний метод `ImageCanvas._zoom`
  і невикористаний у продакшені wrapper `WorkerController.launch_worker`
  видалено; тести життєвого циклу потоків тепер ідуть реально використовуваним
  шляхом `_build_thread`.

  `workers`, `crop`, `canvas`, `widgets`, `settings_dialog`,
  `logging_config`, `main_window`, `app`, `__main__`, `__init__`).
  Запуск через `python -m bgremover` або console-script
  `bgremover`; стара форма `python BgRemover.py` усунена без
  заміни. `BgRemover.py` видалено. Виконано в **13 механічних
  кроках**, кожен зі зеленим тестовим оракулом як шлюзом (140 unit
  + 16 UI тестів, ruff, mypy). Єдина навмисна, поведінково-нейтральна
  зміна коду: `make_tool_icon` тепер розв'язує іконки через
  `importlib.resources` із даних пакета (`bgremover/icons/`) замість
  `__file__`/`sys.argv`/`cwd` — контракт без змін. `pyproject.toml`,
  `Makefile`, CI workflow та macOS-скрипт збірки
  (`create_BgRemover_app.sh`) пристосовані у тому самому зрізі; venv
  встановлює пакет у режимі не-editable (зокрема package-data), тому
  застосунок незалежний від каталогу проєкту.
- Перехідні ре-експорти в `BgRemover.py` (фаза B) та всі імпорти
  `BgRemover` у тестах переведено на пакет у фінальному кроці.

## [2.1.0] – 2026-05-19

### Змінено

- Guard раннього виходу «зображення не завантажено» п'яти методів
  `ImageCanvas` (`apply_round_corners`, `apply_rotate`, `apply_flip`,
  `start_crop_circle`, `start_crop_ratio`) об'єднано в декоратор
  `@_requires_image` – раніше побайтово ідентичний блок зникає;
  поведінка незмінна (захищена наявним набором тестів).
- Фонові worker-и `AIWorker` та `ImageLoadWorker` тепер мають спільний
  базовий клас `_Worker`, який інкапсулює ідентичний потік
  `try/except → logger.exception → error.emit`; підкласи реалізують
  лише `_work()`. `RembgWarmupWorker` свідомо залишається самостійним
  (без сигналу `error`, `finished` завжди у `finally`).
- Зріз версії **2.1.0**: `pyproject.toml` і резервний `__version__` у
  `BgRemover.py` піднято до `2.1.0`; зміни, раніше зібрані під
  `[Unreleased]` (#48/#52/#53, INSTALL_LINUX, раунди 3/4), цим датовано
  як 2.1.0.

### Видалено

- Видалено мертві stylesheet-константи `BTN_STYLE` і `GRP_STYLE`.

### Виправлено

- `save_image()` тепер повідомляє про помилки вводу/виводу як статусне
  повідомлення, а не пропускає їх необробленими.

### Документація

- Додано інструкцію зі встановлення для Linux (`INSTALL_LINUX.md`):
  системні пакети для кожного дистрибутива (apt/dnf/pacman), налаштування venv,
  стартовий скрипт або запис `.desktop` та усунення несправностей; з посиланням у README.
  Включно з особливо простим способом для Raspberry Pi OS (Desktop)
  без venv/pip (PyQt6/Pillow/numpy як системні пакети через `apt`), з
  опційним кроком дооснащення ШІ.

## [2.0.0] – 2026-05-17

Перший задокументований release-стан 2.0.0. У репозиторії немає
історичного Git-тегу `v2.0.0`.

### Функції

- Видалення фону за допомогою ШІ через `rembg` (опційний extra `ai`) включно з
  фоновим прогріванням, щоб перший клік не блокував.
- Інструменти виділення: чарівна паличка (векторизований flood-fill із
  повзунком допуску), пензель, гумка та полігональне ласо; Shift/Ctrl
  для додавання чи віднімання виділення.
- Зробити фон прозорим або замінити кольором.
- Трансформації: обертання (кроки 90° та довільний кут), віддзеркалення,
  заокруглення кутів, обрізання в кількох форматах із сіткою за правилом третин.
- Історія зі скасуванням/повторенням (кнопки на панелі інструментів) і переходом до будь-якого
  попереднього кроку через плаваюче спливне вікно історії.
- Перетягування (Drag & Drop) та «Нещодавно відкриті» (10 записів), обидва через
  асинхронний worker завантаження — без зависання UI.
- Збереження у форматі PNG, JPEG, WebP або TIFF.
- Постійні налаштування (стандартні каталоги, бажаний
  формат файлів) через `QSettings`.
- Збірка App-бандла для macOS (`create_BgRemover_app.sh`) включно з ізольованим
  venv, обробкою Apple Silicon і встановленням іконок; підтримує Python
  3.10–3.15.

### Стабільність і якість

- Захищено потоки worker'ів (немає передчасного GC worker'а,
  чисте завершення потоку в `closeEvent`, race ШІ через монотонний
  лічильник версій полотна).
- Ліміт розміру зображення (40 МП) і захист від decompression bomb під час завантаження.
- Стек скасування з обмеженням пам'яті (256 МБ) з відстеженням байтів за O(1).
- Платформонезалежний шлях журналу (`bgremover.log` у каталозі даних застосунку).
- 108 тестів; `ruff` і `mypy` як кроки CI; CI на Ubuntu та macOS
  з Python 3.10 та 3.12.
- `__version__` зчитується з метаданих пакета (Single Source);
  версія з'являється в заголовку вікна.
