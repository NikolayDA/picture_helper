## 当前状态（2026-07-22，经 Codex 复审后修正）

Ruff、mypy 和本地测试套件仍是新 PR 前的基线。自上一轮以来，**#640–#645** 与 **#648** 已关闭。本更新的早期版本仅依据 2026-07-21 的 `release-abnahme.yml` 派发（提交 `fa2241d`），并错误地将视觉一行（#656）定性为整条链路唯一的阻塞点 —— 一次 PR 复审（Codex）推翻了其中四点，现更正如下：

1. **该次派发证据早已过时。** PR #657（解决 #642，合并于 `521bd63`，晚于 `fa2241d`）将 `waechter_ergebnisse` 设为 `abnahme_aggregate.py::validate_evidence` 中的必填字段，PR #658（解决 #644，合并于 `4416e80`）补充了此前缺失的端到端断言。所引用的派发运行**早于**这两项修复，因此其「✅ 已满足」的各行并不能证明当前代码的状态。在可以将该矩阵作为有效证据引用之前，需要针对当前 `main` 重新触发一次派发。
2. **视觉预评估是咨询性质，并非阻塞项。** `abnahme_aggregate.py::has_blocking_gaps` 明确仅将「Screenshots（视觉预评估）」一行在 `未评估` 时豁免于阻塞（`docs/RELEASE_AUTOMATION.md` 第 4 节：「缺少 `ANTHROPIC_API_KEY` 时，每项标准都保持未评估状态且永不阻塞」）。**#656** 是一项有价值的证据质量改进，但**并非** #646、#639、#595 或 #582 的阻塞项。
3. **Linux x86_64 仍是一个未结标准，而非仅仅是暂停。** 根据 ADR/`RELEASE_AUTOMATION.md` 第 5 节，暂停的 x86_64 硬件冒烟测试在发布决策中被明确视为「声明为未结，未满足」—— 这是一个刻意接受但仍未结的事项，而非已完成的一行。
4. **史诗任务不会自动关闭。** 关闭 #646 只会更新 GitHub 上 #639 的子议题进度；#639、#595 与 #582 各自都需要单独人工复核并关闭。

GitHub 实时状态：**6** 个未结议题。

1. 针对当前 `main`（#657/#658 之后）重新触发一次 `release-abnahme.yml` 派发 —— 此前引用的 2026-07-21 矩阵已不能证明当前代码的状态。

3. 对照 **#646** 自身的验收标准逐一核实（降级容错行为已有证据，且不依赖 #656），满足后即可关闭；随后单独人工复核并关闭 **#639**，接着是 **#595** 与 **#582** —— 任何议题都不会随另一个议题自动关闭。
4. 如需真实的视觉判定结果，可独立处理 **#656** —— 这是质量提升，而非阻塞项。
